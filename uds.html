Session Types

// Key Timers — Critical for Interview
★ Timer Concepts
  • S3 Timer (typically 5s): ECU stays in non-default session only if tester sends TesterPresent (0x3E) before this expires. On timeout → back to Default.
  • P2 (default 50ms): Max time ECU must respond after receiving request.
  • P2* (default 5000ms): Extended wait when ECU sends NRC 0x78 (Response Pending).
  • P3 (default 5s): Minimum gap between end of ECU response and next tester request.
// Interview Questions — 0x10
Q: What happens if TesterPresent is not sent in Extended Session? +
The S3 timer (typically 5 seconds) expires on the ECU side. Once it times out, the ECU automatically transitions back to Default Diagnostic Session (0x01). Any diagnostic services that require Extended Session will then return NRC 0x22 (ConditionsNotCorrect). You must re-send 0x10 0x03 to re-enter the session.
Q: Why does the positive response for 0x10 include P2 and P2* timings? +
The ECU informs the tester of its response timing capabilities in that session. Different sessions may have different P2/P2* values (e.g., Programming Session may have longer P2* since flash operations take time). The tester must use these values to configure its timeout — not rely on default values — to avoid false timeout errors during flash or heavy-computation operations.
Q: Difference between Default Session and Extended Session? +
Default Session: Always active after power-on/reset. Allows basic services like ReadDTCInfo, ReadDataByIdentifier (limited DIDs), TesterPresent. No security-sensitive operations.

Extended Diagnostic Session: Requires explicit 0x10 0x03. Unlocks DTC clearing, I/O control (0x2F), WriteDataByIdentifier (0x2E), SecurityAccess (0x27), and many OEM-specific services. Requires active keep-alive via TesterPresent.

// Interview Questions — 0x11
Q: Difference between Hard Reset (0x01) and Soft Reset (0x03)? +
Hard Reset (0x01): Simulates full power-off/power-on. All volatile memory is cleared. Resets every subsystem. Takes longer to complete. Used after ECU programming to apply new software. NVM (EEPROM/Flash) data is preserved.

Soft Reset (0x03): Software-controlled restart only. No power cut. RAM cleared, application re-initialized, but hardware peripherals may retain state. Faster. Used for minor configuration resets or watchdog-triggered resets.
Q: Why is positive response sent BEFORE the actual reset? +
Because once the ECU resets, it loses communication context and cannot send any CAN frames. The positive response (0x51) is sent to acknowledge the request was received and accepted, then the ECU resets. The tester must wait for the ECU to re-appear on the bus (typically 2-5 seconds) before sending new UDS requests. If the tester doesn't wait and immediately sends a request, it will get no response.
Q: In what sequence do you typically use 0x11 after ECU flashing? +
Typical post-flash sequence: (1) Complete flashing with 0x34/0x36/0x37, (2) Send 0x31 0x01 (CheckMemory routine) to verify CRC, (3) Send 0x31 0x03 (CheckProgrammingPreConditions), (4) Send 0x11 0x01 (Hard Reset) to apply new firmware, (5) Wait 2–5s for ECU boot, (6) Send 0x10 0x01 to confirm Default Session is active, (7) Read SW version via 0x22 to verify flash success.

// Interview Questions — 0x22
Q: Can you read multiple DIDs in one 0x22 request? +
Yes. ISO 14229 allows requesting multiple DIDs in a single 0x22 frame by concatenating the 2-byte DIDs: e.g., 22 F1 90 F1 88 reads VIN and SW version together. The response (0x62) will contain each DID followed by its data in the same order. However, the combined response must fit within the transport layer message size (e.g., 4095 bytes in ISO-TP). If the ECU doesn't support multi-DID read, it returns NRC 0x31.
Q: What is the DID range convention? +
ISO 14229 divides the 2-byte (0x0000–0xFFFF) DID range: 0x0000–0x00FF: ISO SAE reserved. 0xF100–0xF1FF: Standard identification DIDs (VIN, SW version, part number etc.). 0xF200–0xF2FF: Periodic DIDs (for 0x2A service). 0xF300–0xFCFF: Dynamically defined DIDs (via 0x2C service). 0xFD00–0xFEFF: OEM-defined. 0xFF00–0xFFFF: System supplier defined. In practice for interviews, F1xx range DIDs are most commonly tested.
Q: How does 0x22 differ from 0x2A (ReadDataByPeriodicIdentifier)? +
0x22 is an on-demand, single poll — tester requests once, ECU responds once. Used for static data or snapshot reads. 0x2A is a subscription model — tester subscribes to a periodic DID with a transmission rate (slow/medium/fast), and the ECU autonomously sends data at that rate without further requests. 0x2A is used for real-time monitoring of dynamic signals like sensor values, temperatures, RPM during diagnostics.

// Interview Questions — 0x2E
Q: What prerequisite steps are needed before 0x2E can succeed? +
Mandatory sequence: (1) Enter Extended Diagnostic Session: 0x10 0x03. (2) Perform Security Access: 0x27 0x01 (send seed request) → receive seed → compute key → 0x27 0x02 send key → receive 0x67 0x02. (3) Now send 0x2E + DID + data. If Security Access is skipped, ECU returns NRC 0x33. If wrong session, NRC 0x22. The security level required depends on OEM specification.
Q: How do you verify a 0x2E write was successful? +
Two steps: (1) Confirm the positive response 0x6E + echoed DID was received (confirms ECU accepted the write request). (2) Perform a 0x22 read of the same DID to verify the new value is now returned. This is especially important for NVM writes because the 0x6E response only means the write was accepted, not necessarily completed to non-volatile memory. Some ECUs require an ECU reset (0x11) for NVM values to be finalized and persistent.

★ Snapshot Data (0x04) — What Is It?
  • Also called Freeze Frame. ECU stores a snapshot of live sensor values at the moment a DTC was set.
  • Contains: engine speed, coolant temp, vehicle speed, battery voltage, injector duty cycle etc. — exactly what was happening when fault occurred.
  • Request: 19 04 [DTC high] [DTC mid] [DTC low] FF (0xFF = all records).
  • Response: 0x59 04 + DTC + snapshotRecordNumber + dataIdentifier + data values.
  • Critical for root cause analysis — "what was the car doing when this DTC was stored?"
// Extended Data (0x06) — Aging & Occurrence Counters
Extended Data Record Contents
  • Occurrence Counter: How many times this DTC has been set (0–255). Useful for intermittent faults.
  • Aging Counter: Counts up each ignition cycle without the fault. When it reaches aging threshold (e.g. 40 cycles), DTC is auto-healed. Reset to 0 if fault recurs.
  • Aging Threshold: OEM-defined max aging cycles — typically 40.
  • FaultDetectionCounter: Internal debounce counter — goes from 0 to threshold before DTC is confirmed.
// Interview Questions — 0x19 (Most Critical)
Q: What is the difference between a pending DTC and a confirmed DTC? +
Pending DTC (bit 2 of status = 1): The diagnostic test has detected a fault condition for the first time in the current or recent operating cycle, but has not yet met the OEM-defined confirmation threshold. It could be a transient fault. It is stored as "pending."

Confirmed DTC (bit 3 = 1): The fault has been detected across multiple drive cycles or has exceeded the debounce/occurrence threshold defined in the ECU. This triggers MIL illumination in emissions-relevant systems. Confirmed DTCs persist in NVM even after ignition off. Pending DTCs may be cleared automatically if the fault doesn't recur.
Q: What is DTCStatusAvailabilityMask in the 0x59 response? +
This byte in the 0x59 response tells the tester which status bits the ECU actually supports. Not all ECUs implement all 8 bits. For example, if bit 7 (warningIndicatorRequested) is 0 in the availability mask, it means the ECU doesn't track that bit and it will always be 0 in any DTC status. Testers must AND the requested status mask with the availability mask to understand what filtering is actually meaningful. If you request with mask 0xFF but availability mask is 0x2F, bits 6 and 7 are meaningless for filtering.
Q: What is the difference between 0x19 0x02 and 0x19 0x0A? +
0x19 0x02 (reportDTCByStatusMask): Returns only DTCs whose current status matches the provided mask. You can filter for active faults (mask 0x01), confirmed faults (0x08), or pending (0x04). DTCs that don't match the mask are excluded from the response. If no DTCs match, response is 0x59 0x02 + availMask with no DTC records.

0x19 0x0A (reportSupportedDTC): Returns ALL DTCs the ECU can potentially set, regardless of their current status — even DTCs that have never been triggered. Used for DTC catalogue validation and to verify the ECU supports all required DTCs per OEM specification.
Q: How do you clear DTCs and what is the relationship to 0x19? +
DTCs are cleared using service 0x14 (ClearDiagnosticInformation), not 0x19. The flow is: (1) Read DTCs with 0x19 to document them, (2) Send 0x14 + DTC group mask (e.g. 0x14 FF FF FF clears all DTCs), (3) Receive 0x54 positive response, (4) Re-read with 0x19 0x02 to confirm DTCs are cleared. Note: Permanent DTCs (reportable via 0x19 0x17) cannot be cleared by 0x14 — they can only be cleared by the ECU's own internal monitor running successfully for the required number of drive cycles (CARB/OBD requirement).
Q: Explain DTC aging with a practical example? +
Scenario: A wheel speed sensor fault is set (DTC confirmed). After repair, the fault no longer occurs. The aging counter in extended data (0x19 0x06) increments by 1 every ignition cycle where the test runs and passes. OEM threshold = 40 cycles. After 40 fault-free cycles, the ECU automatically removes the confirmed DTC from memory — it has "aged out." If the fault recurs on cycle 25, the aging counter resets to 0 and must restart the count. This mechanism prevents the ECU from retaining stale DTCs forever, and distinguishes genuinely repaired faults from ignored ones.
