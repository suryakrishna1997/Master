Validation & Verification Fundamentals

Verification answers "Are we building the product right?" — checks conformance to specifications. Validation answers "Are we building the right product?" — checks it meets user needs. Embedded V&V adds real-time, concurrency, and hardware-software boundary concerns.

VVerification Techniques

  • Static: code reviews, walk-throughs, lint, MISRA-C, static analyzers (Polyspace, Coverity)
  • Unit tests: function-level, mock hardware dependencies
  • Integration: verify module interfaces & data flow
  • Coverage: MC/DC for safety-critical (DO-178C)
  • Regression: automated, runs on every commit

VValidation Techniques

  • System tests: on target hardware, real peripherals
  • HIL: Hardware-in-the-Loop with plant models
  • SIL: Software-in-the-Loop simulation
  • Field tests: environmental, EMC, longevity
  • Acceptance: against user/contract requirements

RTOS verification focuses on determinism, schedulability, and concurrency correctness. A missed deadline in a hard real-time system is a failure, not a slowdown.

RCore Concepts to Verify

  • Task scheduling: preemptive priority, round-robin
  • Priority inversion: test with high/low priority + shared resource
  • Deadlocks: circular wait on mutexes
  • Starvation: low-priority task never runs
  • ISR latency: time from IRQ to first ISR instruction
  • Context switch time: measured in cycles
  • Timer jitter: variation in periodic task start

RVerification Approaches

  • WCET analysis: worst-case execution time, static + measured
  • CPU load monitoring: idle task run-time, < 70% target
  • Stack watermark: paint pattern, check high-water mark
  • Trace: Segger SystemView, Tracealyzer, ITM/SWO
  • Stress: flood queues, overrun buffers, watch recovery
  • Watchdog: verify kick pattern, timeout behavior
Classic Bug · Priority Inversion

Low-priority task L holds mutex → high-priority task H blocks → medium task M preempts L → H is effectively starved by M. Fix: priority inheritance (mutex) or priority ceiling protocol. Famous case: Mars Pathfinder (1997).

priority_inversion_test.cRTOS
// Test: verify priority inheritance on a mutex
void test_prio_inheritance(void) {
    osMutexId_t mtx = osMutexNew(&(osMutexAttr_t){
        .attr_bits = osMutexPrioInherit  // critical
    });
    create_task(low_task,  PRIO_LOW,  "low");
    create_task(high_task, PRIO_HIGH, "high");
    // low grabs mutex, high preempts & blocks,
       // medium must NOT starve high — measure H's wakeup latency
    TEST_ASSERT_LT(latency_us(), 500);
}

RT-Linux (PREEMPT_RT) converts the standard GPL kernel into a fully preemptive kernel. Verification focuses on worst-case latency and determinism under load.

LPREEMPT_RT Patch

  • Makes kernel fully preemptible (except raw IRQs)
  • High-resolution timers (hrtimer)
  • Threaded IRQ handlers
  • Priority inheritance on rt_mutex
  • Spinlocks → rt_mutexes (sleepable)
  • Check: uname -a contains PREEMPT RT

LLatency Measurement Tools

  • cyclictest: measures scheduling jitter
  • hackbench: stress scheduler
  • ftrace / trace-cmd: kernel function tracing
  • perf: profiling, hardware counters
  • rt-tests suite: pi_stress, signaltest
  • BPF / bpftrace: modern observability

MConcurrency Hazards

  • Race condition: shared data, unsynchronized access
  • Data race: ≥1 write, no synchronization, UB
  • Deadlock: 4 conditions (mutual exclusion, hold & wait, no preemption, circular wait)
  • Livelock: threads active, no progress
  • Lost wakeup: signal before wait
  • Aba: compare-and-swap, value reused

MVerification Approaches

  • ThreadSanitizer (TSan): data race detection
  • Helgrind: Valgrind's error detector
  • Stress tests: millions of iterations
  • Mock random scheduling: chaos testing
  • Formal verification: TLA+, SPIN model checker
  • Coverage with concurrency: line + branch + mutex states
Heisenbug Warning

Concurrency bugs reproduce rarely. Run tests under load, vary CPU counts, inject delays, use stress-ng. A test passing once proves nothing — run it 100k times.

ARM & RISC-V Platforms

Modern embedded computing runs on ARM Cortex-M (microcontroller class), Cortex-A (application class with MMU/Linux), and increasingly RISC-V. Knowing the register model, exception handling, and memory map is essential for low-level debugging and verification.

MCortex-M0/M0+

  • Von Neumann, ARMv6-M
  • Thumb-only instruction set
  • No FPU, no DSP
  • 12 cycles IRQ latency (M0)
  • 3-stage pipeline
  • IoT, ultra-low-power

MCortex-M3

  • Harvard architecture, ARMv7-M
  • Thumb-2 (32-bit instructions)
  • NVIC: 240 IRQs, 256 priorities
  • 12-cycle IRQ latency
  • MSP + PSP stack pointers
  • SysTick timer standard

MCortex-M4/M7

  • ARMv7E-M, DSP instructions
  • Optional single/double FPU
  • M7: 6-stage superscalar, L1 cache
  • TCM (tightly-coupled memory)
  • MPU (M4) / MPU + cache (M7)
  • Audio, motor control, vision

Exception Model

Cortex-M has 16 system exceptions (Reset, NMI, HardFault, MemManage, BusFault, UsageFault, SVCall, PendSV, SysTick, etc.) + N external IRQs. Vector table at 0x00000000 (or VTOR). Each entry is 4 bytes (address). HardFault is the catch-all for unrecoverable errors — debug with CFSR/HFSR registers.

hardfault_handler.cCortex-M3/M4
// Capture fault state for post-mortem analysis
__attribute__((naked)) void HardFault_Handler(void) {
    __asm volatile(
        " tst lr, #4                \n"  // MSP or PSP?
        " ite eq                    \n"
        " mrseq r0, msp             \n"
        " mrsne r0, psp             \n"
        " ldr r1, =hardfault_dump   \n"
        " bx r1                     \n"
    );
}

void hardfault_dump(uint32_t *sp) {
    volatile uint32_t r0,r1,r2,r3,r12,lr,pc,psr;
    r0=sp[0]; r1=sp[1]; r2=sp[2]; r3=sp[3];
    r12=sp[4]; lr=sp[5]; pc=sp[6]; psr=sp[7];
    printf("PC=0x%08lx LR=0x%08lx PSR=0x%08lx\n", pc, lr, psr);
    printf("CFSR=0x%08lx HFSR=0x%08lx\n", SCB->CFSR, SCB->HFSR);
    while(1);  // halt for debugger
}

ACortex-A Family

  • A53/A55: 64-bit, low-power, big.LITTLE little
  • A72/A76/A78: high-performance 64-bit
  • A32/A35: 32-bit entry-level
  • Full MMU → runs Linux, Android
  • VFPv4 + NEON SIMD standard
  • TrustZone security extension
  • L1 + L2 caches, often L3

APrivilege & Modes

  • EL0: User (applications)
  • EL1: Kernel (OS)
  • EL2: Hypervisor (virtualization)
  • EL3: Secure monitor (TrustZone)
  • AArch32 modes: USR, FIQ, IRQ, SVC, ABT, UND, SYS
  • Banked registers per mode
  • CP15 for system control
Debugging Cortex-A

Use JTAG with OpenOCD. Boot sequence: BootROM → BL1 (SPL) → BL2 → BL31 → U-Boot → Kernel. Each stage is a debugging boundary. devmem tool lets you peek/poke physical memory from userspace. /proc/iomem shows kernel memory map.

VRISC-V ISA Basics

  • Open ISA, no licensing fees
  • Base: RV32I (32-bit, 37 instructions)
  • Standard extensions: M (mul/div), A (atomic), C (compressed 16-bit), F/D (float), V (vector)
  • Common configs: RV32IMAC, RV64GC
  • 31 general-purpose registers (x0 always 0)
  • Load-store architecture
  • No condition codes — uses explicit compare-and-branch

VPrivilege Levels

  • M-mode: Machine — full access, mandatory
  • S-mode: Supervisor — runs OS kernel
  • U-mode: User — applications
  • CSR: Control & Status Registers
  • PMP: Physical Memory Protection
  • CLINT (timer) + PLIC (interrupt controller)
  • OpenSBI handles M-mode runtime services

Pipeline Hazards

Data hazards: RAW, WAW, WAR — resolved by forwarding/stalls. Control hazards: branches flush pipeline — branch prediction (M7) mitigates. Structural hazards: resource conflicts. Memory wait states on flash → insert wait cycles or use cache/TCM. For timing-critical code, place in TCM (Tightly Coupled Memory) for zero-wait-state execution.

UKey Characteristics

  • Async: no clock line, both ends agree baud
  • Full-duplex: TX + RX + GND
  • Topology: point-to-point (or RS-485 multi-drop)
  • Baud rates: 9600, 19200, 38400, 57600, 115200, 460800, 921600, 3M+
  • Frame: Start(1) + Data(5–9) + Parity(optional) + Stop(1/1.5/2)
  • Levels: TTL (0/3.3V or 0/5V), RS-232 (±12V), RS-485 (differential)

UTesting UART

  • Loopback: tie TX→RX, send & verify
  • Baud mismatch: frame errors (FE bit in ISR)
  • Noise: NE bit, overrun (ORE)
  • Stress: burst tx, fill FIFO, test backpressure
  • Multi-drop (RS-485): direction control timing
  • Scope check: bit width = 1/baud seconds
Baud Rate Tolerance

Receiver samples each bit at its midpoint. Cumulative drift across 10 bits must be < ±2.5%. Use high-precision crystal or auto-baud detection. Common pitfall: APB clock divisor rounding causes 1.5% error → works at 9600, fails at 921600.

IKey Characteristics

  • Sync: SCL clock + SDA data
  • Half-duplex: single data line
  • Multi-master: arbitration + clock sync
  • Addressing: 7-bit (128 devs) or 10-bit
  • Speeds: Std 100 kHz · Fast 400 kHz · Fast+ 1 MHz · HSM 3.4 MHz
  • Open-drain: requires pull-up resistors (typ 4.7kΩ)

ITesting I²C

  • Address scan: sweep 0x03–0x77, log ACKs
  • Rise time: check with scope (≤ 300ns @ 1MHz)
  • Pull-up sizing: Rp = tr / (0.8473 × Cb)
  • Bus stuck: clock 9 cycles to release
  • Arbitration test: two masters simultaneously
  • Clock stretching: slave holds SCL low
Common I²C Bug

A device gets stuck holding SDA low (mid-transfer crash). Recovery: manually toggle SCL 9 times, then issue a STOP. Always implement I²C bus recovery in production firmware.

  • Speed: up to 50+ MHz (device-limited)

STesting SPI

  • Loopback: MOSI→MISO, send pattern, verify
  • Mode match: CPOL/CPHA must match slave
  • CS timing: setup/hold vs SCK
  • Signal integrity: ringing, overshoot at high speeds
  • Clock polarity: verify idle state on scope
  • Multi-slave CS: ensure only one active

CKey Characteristics

  • Differential: CAN_H, CAN_L (twisted pair)
  • Multi-master: any node can transmit

CTesting CAN

  • Arbitration: lower ID wins — verify with collisions
  • Bit timing: set sample point ~75–87.5%
  • Termination: measure 60Ω across bus (two 120Ω in parallel)
  • BUS-OFF recovery: induce errors, verify re-entry
  • Listen-only mode: passive monitoring
  • Tools: can-utils, PCAN-View, Vector CANalyzer
CAN Bus-Off State

When a node's Transmit Error Counter (TEC) exceeds 255, it goes BUS-OFF — disconnected from bus. Recovery requires software intervention (or auto-recovery after 128 × 11 consecutive recessive bits). Verify your firmware handles this gracefully without stranding the node.

DDigital I/O Testing

  • Input debounce: RC filter + software, verify with noisy switch
  • Edge detection: EXTI rising/falling/both
  • Output modes: push-pull, open-drain
  • Pull resistors: internal pull-up/down — verify value (typ 40kΩ)
  • Slew rate: check edges with scope
  • Drive strength: source/sink current limits
  • Boundary scan: JTAG boundary scan cells (IEEE 1149.1)

AAnalog I/O Testing

  • ADC resolution: 8/10/12/16/24-bit
  • Sampling rate: verify effective vs spec
  • Reference voltage: internal, external, VDD
  • Linearity: DNL (differential), INL (integral)
  • Offset & gain error: calibrated out
  • Noise: short input, measure RMS
  • DAC: monotonicity, settling time, glitch energy
  • PWM: duty cycle accuracy, frequency stability
Practical Tip

For ADC testing, use a precision voltage source (or calibrated DAC) and sweep across the full range. Plot INL — large deviations indicate ADC non-linearity or reference noise. Always test with the actual signal source impedance; high source impedance causes sampling errors due to insufficient acquisition time.

Checkpoint
3. On a CAN bus, two nodes transmit simultaneously: ID 0x100 and ID 0x200. Which wins arbitration?
A0x100 (lower ID = higher priority)
B0x200 (higher ID = higher priority)
CBoth retry after random delay
DWhichever started first
CAN arbitration is non-destructive: dominant bit (0) wins over recessive (1). Lower ID has a 0 in a higher bit position earlier, so 0x100 wins. The losing node stops transmitting when it sees a dominant bit where it wrote recessive.

Linux HMI Testing & Scripting

Human-Machine Interface devices on Linux range from small e-ink displays to full touch panels running Qt. Verification covers UI rendering, input response, performance stability, and the shell tooling that automates test execution on the target.

HHMI Test Areas

  • Rendering: frame rate, tearing, partial refresh (e-paper)
  • Touch: accuracy, multi-touch, latency < 100ms
  • Visuals: colors, contrast, brightness under sunlight
  • State transitions: every screen reachable, returns cleanly
  • Input validation: special chars, max-length, ranges
  • Internationalization: RTL, character rendering, fonts
  • Accessibility: contrast ratios, focus indicators

HPerformance & Stability

  • Boot time: splash to interactive < spec
  • Memory: valgrind --tool=massif, watch for leaks over 24h
  • CPU: top, htop, idle should be > 80%
  • Watchdog: system kick under load
  • Disk: write endurance, log rotation
  • Power: sleep/wake cycles, deep sleep current
  • Thermal: long-running UI stress, thermal throttling

Hardware Test Setup & Troubleshooting

Bench equipment and test fixtures turn firmware bugs into observable signals. A well-equipped setup with proper probing technique is the difference between finding a 50 ns glitch in an hour versus never finding it at all.

DDigital Multimeter

  • Voltage (DC/AC): power rails, signal levels
  • Current: sleep, active, peak (µA range matters)
  • Resistance: pull-ups, terminations, traces
  • Continuity: beep mode for shorts/opens
  • Diode test: forward voltage drop ~0.6V
  • Capacitance: basic check
  • Beware: burden voltage on current ranges

OOscilloscope

  • Bandwidth: ≥ 5× signal freq (e.g. 50 MHz for 10 MHz)
  • Sample rate: ≥ 5× bandwidth (Nyquist is minimum)
  • Probes: 10× to reduce loading (10 MΩ, ~10 pF)
  • Triggers: edge, pulse width, runt, glitch, protocol
  • Math: add, subtract, FFT, decode (I²C/SPI/UART/CAN)
  • Coupling: AC removes DC component; DC shows truth
  • Memory depth: captures long bursts at high rate

LLogic Analyzer

  • Channels: 8–34 typical, parallel capture
  • Sampling: 100 MS/s+ for high-speed buses
  • Protocol decode: I²C, SPI, UART, CAN, 1-Wire, USB
  • Trigger: complex multi-byte patterns
  • Software: Saleae Logic, Sigrok/PulseView
  • Limitation: voltage only, not analog quality
  • Use: parallel buses, multi-signal timing

PPower Supply

  • Voltage + current limits: never exceed rail spec
  • Current limit: catch shorts before damage
  • Sequencing: multi-rail ordering (1.2V core → 3.3V I/O)
  • Measurement: read-back current for power profiling
  • Battery sim: variable source + sink for charging tests
  • E-Load: programmable load for regulator stress

JTest Jig / Fixture

  • Pogo pins: spring contacts to test pads
  • Bed of nails: PCB-level production testing
  • Boundary scan: JTAG (IEEE 1149.1) for pins
  • Modular: quick swap of DUT boards
  • Automation: pneumatic actuation, camera inspection
  • Calibration: verify fixture itself periodically
Probing Pitfalls

Ground lead inductance: the 6-inch alligator clip acts as an inductor; at high frequencies it causes ringing. Use a ground spring tip for >20 MHz signals. Probe loading: a 10 pF probe on a 1 MHz SPI clock shifts edge timing — use 10× mode. Floating measurements: never connect a grounded scope probe to mains or non-isolated circuits; use a differential probe.

Troubleshooting Method

1. Reproduce reliably — nothing beats a deterministic repro. 2. Bisect: what's the smallest config that fails? 3. Divide & conquer: is it HW or SW? Swap boards, swap firmware. 4. Measure first, theorize second — assumptions waste hours. 5. Document findings — future-you will thank present-you.

6. You're probing a 20 MHz SPI clock with a scope and see ringing. What's the first thing to change?

BIncrease the scope bandwidth limit

The standard alligator ground clip's inductance causes ringing at high frequencies. A ground spring tip removes most of the lead length, dramatically reducing parasitic inductance. 1× probes add capacitance and make it worse.

Flashing, Debugging & Log Analysis

From initial bring-up to field failure analysis, you live in debuggers and logs. Knowing the toolchain — flashing methods, debug probe protocols, log analysis techniques — turns hours of guesswork into minutes of evidence-based diagnosis.

FFlashing Methods

  • SWD: 2-wire (SWCLK + SWDIO), Cortex-M standard
  • JTAG: 4-wire (TCK/TMS/TDI/TDO) + TRST, more boundary-scan
  • UART bootloader: ROM bootloader, no probe needed
  • USB DFU: Device Firmware Upgrade class
  • ISP: In-System Programming, SPI/I²C/SPI-flash
  • OTA: Over-The-Air, A/B partitioning for safety
  • OTP/eFuse: one-time programmable, careful!

PDebug Probes

  • ST-Link: ST-MCU focused, cheap, ubiquitous
  • J-Link: Segger, fast, broad vendor support
  • CMSIS-DAP: open standard, low-cost
  • Probe-rs: open-source, modern, Rust-based
  • Lauterbach Trace32: high-end, JTAG + trace
  • Black Magic: native GDB server, open HW
  • DAPLink: drag-drop flashing via USB MSD

GGDB + OpenOCD

  • OpenOCD: translates GDB ↔ probe protocol
  • arm-none-eabi-gdb: cross-debugger
  • Break on hardware address: hbreak
  • Watchpoint: watch var, data-dependent break
  • Live variable view: watch in TUI mode
  • SWO/ITM: printf-style trace over single pin
  • ETM: full instruction trace (Cortex-M7+)

LLog Analysis

  • Levels: DEBUG/INFO/WARN/ERROR/FATAL
  • Format: timestamp + level + module + msg
  • Ring buffer: in-RAM, dumps on fault
  • journalctl: structured, filterable
  • syslog: network-aggregated
  • ELK/Loki: centralized, searchable
  • Correlation IDs: trace request across modules

Printf vs Step-Debug

Step debugging finds logic bugs fast. Printf finds timing-sensitive bugs that disappear when paused. For hard real-time issues (race conditions, ISR latency), ITM trace is the gold standard — non-intrusive, time-stamped, post-processable. Use the right tool for the bug.
