1. Automotive Networking Fundamentals

Core principles of vehicle bus networks, domain controllers, zonal network layouts, and protocol properties.

Foundations

ECU Communication Architecture

In modern vehicles, an Electronic Control Unit (ECU) is an embedded microcontroller node handling specific systems (e.g. Engine, Braking, Infotainment). Historically, point-to-point wiring (connecting every sensor directly to every actuator) created massive wiring harness weights and cost limitations. To address this, shared serial networks (buses) were introduced. Modern vehicles have evolved from simple decentralized structures into Domain Controller Architectures (grouping ECUs logically like Powertrain, ADAS, Body) and are transitioning to Zonal Architectures (grouping ECUs geographically by physical zone, linked by high-bandwidth Gigabit Ethernet backbones to central compute brains).

Protocol Bus Maximum Data Rate Network Topology Primary Automotive Applications
LIN 20 kbps Master-Slave (Single wire) Body controls, mirrors, window lifters, seat adjusters.
CAN (Classic) 1 Mbps Multi-Master Bus (Differential pair) Powertrain, chassis control loops, basic diagnostic interfaces.
CAN FD 5 - 8 Mbps Multi-Master Bus (Differential pair) ADAS radar links, high-speed gateways, flashing payloads.
FlexRay 10 Mbps Point-to-Point / Star topologies Steer-by-wire, brake-by-wire, safety-critical real-time paths.
MOST 150 Mbps Optical / Copper Ring topology Legacy infotainment streaming, camera projections, amplifiers.
Automotive Ethernet 100 / 1000 Mbps Switched Star topology (UTP cable) ADAS camera feeds, FOTA downloads, LIDAR networks, backbone links.

3. CAN Protocol Complete Guide (ISO 11898)

Physical layer differential signaling, bus arbitration, bit timing layouts, signal endianness formats, and CAN frame bits.

CAN Physical & Data Link Layers

Classic CAN operates over a twisted-pair copper cable using differential signaling to resist electromagnetic noise.

  • Recessive State (Logical 1): Both CAN_H and CAN_L sit at approximately 2.5V (difference is 0V).
  • Dominant State (Logical 0): CAN_H rises to 3.5V, and CAN_L drops to 1.5V (difference is 2.0V). The dominant bit overcomes recessive bits on the bus.
  • Termination Resistors: A 120-ohm resistor is placed at each end of the bus to prevent signal reflections.

CAN 11-Bit standard frame bits

  • SOF (Start of Frame): 1 dominant bit indicating the start of transmission.
  • Identifier: 11 bits specifying message priority and data definition.
  • RTR (Remote Transmission Request): 1 bit (dominant for data frames, recessive for remote requests).
  • IDE (Identifier Extension): 1 dominant bit specifying standard 11-bit ID.
  • DLC (Data Length Code): 4 bits specifying payload byte count (0 to 8).
  • DATA: 0 to 8 bytes payload.
  • CRC: 15 bits cyclic redundancy check for error detection, followed by a 1-bit delimiter.
  • ACK (Acknowledgment): 1 bit (transmitter sends recessive; receiving nodes overwrite with dominant to confirm validation).
  • EOF (End of Frame): 7 recessive bits.

CAN Extended Frame (29-Bit) Details

The Extended frame uses a 29-bit identifier by setting the **IDE** bit to recessive (logical 1) and introducing an 18-bit identifier extension after the **SRR (Substitute Remote Request)** bit, supporting complex message mappings in large networks.

4. CAN FD (Flexible Data-rate) Complete Guide

Why CAN FD exists, payload sizing options up to 64 bytes, and bit rate switching mechanics.

CAN vs CAN FD

Classic CAN is limited to an 8-byte payload and a 1 Mbps speed limit. High bus loads caused by modern ECUs (like ADAS and active gateways) require a higher bandwidth protocol.

CAN FD (ratified in ISO 11898-1:2015) addresses this by:

  • Extended Payloads: Supports data lengths of 8, 12, 16, 20, 24, 32, 48, and 64 bytes.
  • Dual Bit Rate: The arbitration phase operates at standard rates (e.g. 500kbps to 1Mbps), while the data phase switches to higher speeds (e.g. 5Mbps) during payload transmission.

Control Field Flags

  • FDF (FD Format Indicator): Set to recessive (1) to declare CAN FD frame type.
  • BRS (Bit Rate Switch): Set to recessive (1) to trigger transition to the data phase bit speed.
  • ESI (Error State Indicator): 1 bit indicating node error state (dominant = error active, recessive = error passive).

5. CAN Transport Protocol (ISO 15765-2)

Segmentation, reassembly, Flow Control parameters, and frame sequence traces.

Frame Typologies

CAN frames are limited to 8 bytes (or 64 bytes in CAN FD), but UDS payloads (like firmware files) can exceed thousands of bytes. **CAN-TP** manages the transport layer segmentation and reassembly.

  • Single Frame (SF): Used for payloads up to 7 bytes. Byte 0 specifies frame type `0x0` and length.
  • First Frame (FF): Sent first for segmented transfers. Byte 0-1 specifies type `0x1` and the total payload length.
  • Flow Control (FC): Receiver responds with flow parameters (type `0x3`): **FlowStatus** (CTS, WAIT, OVERFLOW), **BlockSize (BS)**, and **STmin** (separation time between subsequent frames).
  • Consecutive Frame (CF): Transmits the segmented payload. Byte 0 type is `0x2` with a 4-bit Sequence Number (SN).

Timing Parameters

  • N_As: Time for transmitter to send a frame on the physical layer.
  • N_Bs: Time between transmitter sending FF and receiving the FC frame (Timeout = 1000ms).
  • N_Cs: Time between transmitter sending CF and receiving the next CF slot.

6. UDS (ISO 14229) & ECU Reset Sequences

Detailed service configurations, positive/negative responses, security Seed-Key flows, DTC masks, and bootloader programming timelines.

UDS Architecture Overview

Unified Diagnostic Services (UDS) is the standard application layer protocol for automotive diagnostics. It defines service request and response formats. Requests start with a **Service Identifier (SID)**. Responses return `SID + 0x40` for positive results, or a **Negative Response Frame (7F SID NRC)** when an error occurs.

UDS Service 0x19: ReadDTCInformation - Deep Dive

UDS Service 0x19 is the primary method for extracting fault diagnostic data from an ECU. Because DTCs track fault states, severity levels, occurrence counters, and operating conditions, Service 0x19 is highly versatile, using various sub-functions to access different information layers.


1. The 1-Byte DTC Status Mask Structure

Every DTC stored in the ECU is associated with a status byte. Understanding this status byte is essential for interpreting UDS 0x19 responses:

Bit Pos Status Name Description
Bit 0 testFailed The most recent test run failed (fault is currently active).
Bit 1 testFailedThisOperationCycle The fault has failed at least once during the current driving cycle.
Bit 2 pendingDTC The fault is active in the current cycle but not yet confirmed.
Bit 3 confirmedDTC The fault has been verified across multiple cycles and is stored in NVRAM.
Bit 4 testNotCompletedSinceLastClear The diagnostic test has not finished running since the last clear command.
Bit 5 testFailedSinceLastClear The diagnostic test has failed at least once since DTCs were cleared.
Bit 6 testNotCompletedThisOperationCycle The test has not completed during the current driving/operating cycle.
Bit 7 warningIndicatorRequested The ECU requests a warning indicator (e.g. Check Engine MIL lamp) to be illuminated.

2. Key Sub-functions (Basics to End)

  • 0x01 - reportNumberOfDTCByStatusMask: Returns the total count of DTCs that match a provided status mask. Extremely basic check to see if any faults are present.
    Request: 19 01 [StatusMask] → Response: 59 01 [StatusAvailabilityMask] [DTCFormat] [CountHigh] [CountLow]
  • 0x02 - reportDTCByStatusMask: Returns a list of DTC numbers along with their status bytes matching the mask.
    Request: 19 02 [StatusMask] → Response: 59 02 [StatusAvailabilityMask] [DTC1_H] [DTC1_M] [DTC1_L] [Status1] ...
  • 0x04 - reportDTCSnapshotRecordByDTCNumber: Retrieves environmental sensor data (freeze frames) captured when the fault first occurred (e.g., engine RPM, coolant temperature, vehicle speed).
    Request: 19 04 [DTC_H] [DTC_M] [DTC_L] [RecordNumber]
  • 0x06 - reportDTCExtendedDataRecordByDTCNumber: Retrieves supplementary diagnostic information such as fault occurrence counters, aging counters, update status, and cycle timers.
    Request: 19 06 [DTC_H] [DTC_M] [DTC_L] [RecordNumber]
  • 0x0A - reportSupportedDTC: Lists all DTCs that the ECU is programmed to monitor, regardless of whether they are currently active or not. Helpful to build diagnostic databases.
    Request: 19 0A → Response: 59 0A [StatusAvailabilityMask] [DTCs...]
  • 0x1A - reportSupportedDTCExtendedDataRecord: Lists all the extended data records supported by the ECU. Represents the end of standard DTC introspection features.
    Request: 19 1A

7. AUTOSAR Core Diagnostics Layer

Classic and Adaptive modules, DEM/DCM interfaces, and PDU router flow layouts.

AUTOSAR Layer mapping

Inside an AUTOSAR ECU stack, diagnostic services are partitioned into modular layers:

  • DCM (Diagnostic Communication Manager): Manages diagnostic sessions (0x10), security levels (0x27), and service request routing (0x22/0x2E).
  • DEM (Diagnostic Event Manager): Processes DTC storage, freeze frames, and fault conditions reported by software components.
  • NvM (Non-Volatile Memory Manager): Saves persistent DTC values and configuration blocks to EEPROM/Flash.
  • PduR (PDU Router): Routes diagnostic payload data packets between L2CAP/CAN-TP buffers and the DCM.

10. Vector Software Toolchain & CAPL

CANoe, CANalyzer configuration, and advanced CAPL diagnostic scripting syntax.

CAPL Diagnostic Scripting Example

CAPL (CAN Access Programming Language) scripting is used inside Vector CANoe configurations to simulate ECUs, inject test patterns, and automate diagnostic validation loops.

/* Simulated ECU responding to Tester Present */
variables
{
  message 0x18DAF110 responseMsg;
}

on message 0x18DA10F1
{
  // Check if first payload byte is Tester Present (0x3E)
  if (this.byte(1) == 0x3E)
  {
    responseMsg.dlc = 8;
    responseMsg.byte(0) = 0x02; // SF Length
    responseMsg.byte(1) = 0x7E; // Positive Response (0x3E + 0x40)
    responseMsg.byte(2) = this.byte(2); // Sub-function
    responseMsg.byte(3) = 0x55; responseMsg.byte(4) = 0x55;
    
    output(responseMsg); // Send frame on the bus
    write("CAPL: Returned positive response for Tester Present");
  }
}
