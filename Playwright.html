The SDET Roadmap

🟢 Module 1: The Basics

Get up and running with Playwright in minutes.

  • Setup: npm init playwright@latest (generates tests, config, GitHub Actions).
  • Locators: Avoid XPath/CSS when possible. Use user-centric locators: getByRole, getByText, getByTestId.
  • Auto-waiting: Playwright waits for elements to be attached, visible, and stable before acting. No more `Thread.sleep()`!
  • Assertions: Built-in web-first assertions (expect(locator).toBeVisible()). It retries until the condition is met.
test('basic test', async ({ page }) => {
  await page.goto('https://playwright.dev/');
  await expect(page).toHaveTitle(/Playwright/);
});

🟡 Module 2: Architecture & Patterns

Scaling up your test automation framework.

  • Architecture: Unlike Selenium (HTTP requests), Playwright communicates over the Chrome DevTools Protocol (CDP) using a single WebSocket connection per browser context. This makes it incredibly fast and bi-directional.
  • Page Object Model (POM): Abstracting UI elements and actions into classes to reduce duplication.
  • Browser Contexts: Think of it as an "Incognito Window". Tests are isolated in milliseconds without launching a new browser.
  • Fixtures: The modern alternative to `beforeEach/afterEach`. Inject dependencies cleanly into your tests.

🔴 Module 3: 5 YOE Masterclass

What sets a Senior SDET apart from the crowd.

  • Global Auth Setup: Don't login via UI before every test. Login once, save the storage state (cookies/local storage), and inject it globally via setup project in playwright.config.ts.
  • Network Interception: Mock backend responses to test frontend resilience. await page.route('**/api/data', route => route.fulfill({json: data})).
  • Custom Fixtures: Extend built-in fixtures to automatically log in, set up database state, or provide API clients to tests.
  • Sharding & CI/CD: Run 1000s of tests in minutes by splitting the test suite across multiple CI machines using --shard=1/4. Merge reports seamlessly.
  • Tracing & Debugging: Trace Viewer is a game changer. It captures a DOM snapshot of every action, network requests, and console logs.

⚙️ Module 4: Advanced Automation Patterns

Beyond the basics – battle‑tested patterns for large‑scale projects.

  • Visual Regression: Use expect(page).toHaveScreenshot() to catch unintended UI changes.
  • Component Testing: Test UI components in isolation using Playwright's component testing (React, Vue, Svelte).
  • Parallelization: Run tests on multiple browsers simultaneously with projects in config.
  • Service Workers: Mock network responses at the browser level for offline/error scenarios.
  • Custom reporters: Build a JSON/HTML reporter with test metadata for dashboards.
// playwright.config.ts – parallel projects
projects: [
  { name: 'chromium', use: { ... } },
  { name: 'firefox', use: { ... } },
  { name: 'webkit', use: { ... } }
]

Selenium uses the WebDriver protocol (HTTP requests). Every command creates a new HTTP request, leading to overhead and flakiness.

Cypress runs inside the browser loop. It has deep access but is limited to a single tab/origin and struggles with multi-page/multi-tab scenarios.

Playwright uses the Chrome DevTools Protocol (CDP) (and equivalent for WebKit/Firefox) over a WebSocket connection. This allows for bi-directional communication, out-of-process execution (no single-origin limit), multi-tab support, and native network interception.

A senior engineer never logs in via the UI for every test. That adds 5-10 seconds per test. Instead:

  1. Create a global.setup.ts or a setup project in the config.
  2. Perform the UI or API login once.
  3. Save the state: await page.context().storageState({ path: 'auth.json' }).
  4. In playwright.config.ts, configure all test projects to use use: { storageState: 'auth.json' }.

For role-based testing (Admin vs User), I create separate storage states and use custom fixtures to inject the correct context based on the test needs.

Fixtures establish an isolated environment for each test. Hooks like beforeEach rely on global variables/state mutation, which leads to test pollution.

Fixtures provide Encapsulation and Lazy Loading. If a test doesn't ask for a fixture, the setup code doesn't run. With fixtures, you can encapsulate Page Objects, API clients, or database states directly into the test parameters.

export const test = base.extend({
  adminPage: async ({ page }, use) => {
    await loginAsAdmin(page);
    await use(new AdminPage(page)); // test runs here
    await cleanUpAdminData(); // teardown
  }
});

Flakiness is usually caused by race conditions or environmental instability. My approach:

  • Never use hard waits (`waitForTimeout`). Always use web-first assertions (expect().toBeVisible()) which retry automatically.
  • Wait for API responses: Instead of waiting for a UI element, wait for the underlying network call: await Promise.all([page.waitForResponse('**/api/data'), page.locator('btn').click()]).
  • Strict Locators: Ensure locators uniquely identify elements. Playwright throws if a locator is not strict.
  • Trace Viewer: Enable traces on first retry in CI. If a test fails, the trace viewer shows the exact DOM state, console, and network logs.

Network interception allows you to mock API responses, abort requests (like tracking scripts), or modify headers.

Use Cases:

  • Edge cases: Testing UI behavior when the server returns a 500 error or empty list, without having to actually break the backend.
  • Speed: Mocking slow 3rd party APIs (like payment gateways).
  • Blocking: await page.route('**/*.{png,jpg,jpeg}', route => route.abort()); to speed up tests by not loading images.

Playwright supports native Sharding. If I have 1000 tests, I don't run them sequentially. I spin up 5 workers in GitHub Actions (or Jenkins).

Worker 1 runs: npx playwright test --shard=1/5
Worker 2 runs: npx playwright test --shard=2/5

After all jobs complete, I use Playwright's merge-reports tool to combine the HTML reports into a single, unified artifact. This reduces a 1-hour test run to 12 minutes.

Playwright provides an APIRequestContext via the request fixture. I use it for fast test setup and teardown.

Instead of clicking through the UI to create a user (which takes 10 seconds), I hit the POST API endpoint directly to create the data in 200ms. Then, I only use the UI for the specific action I am testing.

You can also use the request fixture to assert database state after a UI action.

POM is standard, but at scale, it can lead to massive "God Classes". For 5 YOE, I introduce:

  • Component Objects: Instead of a whole page, I model reusable components (e.g., a DatePicker class) that can be imported into multiple Page Objects.
  • Fluent Interfaces: Returning `this` from methods to allow chaining: loginPage.enterUser().enterPass().submit().
  • Fixtures for Composition: Using Playwright fixtures to automatically inject Page Objects into tests, reducing boilerplate instantiations (new LoginPage(page)).

Use environment variables and configuration files per environment (dev, staging, prod).

  • Store base URLs, API keys, and credentials in .env files (never in code).
  • Use Playwright's baseURL in config to keep tests environment-agnostic.
  • For test data, use factory functions or seed a test database via API before the test run.
  • Always clean up after tests using afterEach hooks or fixture teardown to avoid data pollution.

Playwright supports device emulation out of the box. In your config, define projects for different devices:

  • Use devices['iPhone 12'] or custom viewports.
  • Test touch events, orientation changes, and mobile-specific UI.
  • Combine with network throttling to simulate poor network conditions.
  • Run these in parallel with desktop projects using sharding.
// playwright.config.ts
import { devices } from '@playwright/test';

projects: [
  {
    name: 'iPhone 12',
    use: { ...devices['iPhone 12'] }
  }
]
